# AI 接口稳定性规范

## 目标
在第三方 AI 服务存在波动时，依然保证接口行为可预测、错误可定位、前端体验可恢复。

## 何时使用
- 新增或修改 `/api/ai/*` 接口
- 接入新的 AI 服务商或模型
- 调整异步任务轮询逻辑
- 处理超时、失败重试、错误提示文案

## 服务端规范
1. 入口校验必须使用 `zod` + `@hono/zod-validator`。
2. 环境变量缺失要显式报错，禁止静默失败。
3. 对第三方响应做结构化解析，不直接信任返回字段。
4. 异步任务必须有：轮询间隔、最大次数、超时错误。
5. 返回格式统一：成功 `{ data }`，失败 `{ error }`。

## 前端规范
1. 使用 typed hono client + React Query mutation/query。
2. `response.ok` 为 false 时，解析错误体并抛出明确错误。
3. UI 必须处理 `isPending`、`isError`、`onSuccess` 三种状态。
4. 错误提示优先展示后端返回信息，兜底通用文案。

## 错误分层
- 配置错误：API Key 缺失、环境变量错误
- 参数错误：输入不合法、图片 URL 非法
- 任务错误：轮询超时、任务失败、结果缺失
- 网络错误：请求失败、第三方服务不稳定

## 检查清单
- 是否覆盖了 API Key 缺失场景
- 是否限制了最大轮询次数
- 是否有超时错误返回
- 前端是否对失败场景给出可理解提示
- 是否避免把第三方原始异常直接暴露给用户

## 输出格式
应用本 skill 时按以下结构汇报：
1. 接口变更点
2. 容错与超时策略
3. 前端错误处理改动
4. 手动验证场景
